# Life Timeline — Week Grid View (Addendum)

This document extends `life-timeline-plan.md`. It defines the week grid view as a distinct feature with its own phase. All existing phases are unchanged. The grid shares the app's data fetch, card system, and CSS token system.

---

## Week Grid View

### Mental Model

A rectangular grid where each row is a calendar year and each column is an ISO week number. Cells are coloured by the most specific active span at that point in time. The grid answers "what was I doing with my life?" at a glance — it is a summary view, not an event browser. Detail is accessed by clicking.

This view replaces the Year zoom level on the subway map. It is activated by selecting Year zoom on the existing zoom control; the subway map's Day and Month zoom levels are unchanged. A separate "Map / Grid" toggle is not needed.

---

## Layout

- **Rows:** one per calendar year, birth year to current year inclusive
- **Columns:** ISO week numbers 1–52 or 1–53 (varies by year)
- **Birth year row:** cells before the birth week render as empty/neutral — no data, not clickable
- **Current year row:** cells after the current ISO week are not rendered
- **Row labels:** year number on the left edge of each row
- **Column labels:** sparse week-number markers every 13 weeks (~quarterly) to give orientation without clutter
- **No virtualisation needed** — a 35-year grid is ~35 rows, a few hundred pixels tall at most

---

## Cell Colour Logic

Each cell is coloured by the highest-priority active span covering that week. Priority order, highest first:

| Priority | Family | Colour source |
|---|---|---|
| 1 | Travel | `base_color_hsl` from the `travel` family definition |
| 2 | Employment | `base_color_hsl` from the `employment` family definition |
| 3 | Education | `base_color_hsl` from the `education` family definition |
| 4 | Residence | Programmatically derived from the location label string (see below) |
| — | No data | `--color-surface` token (neutral) |

Neutral cells (no known residence yet) appear only in early childhood before the first spine relocation event. Once a residence is established it carries forward until the next relocation event.

### Concurrent Travel Spans

If two travel spans overlap a single week, the one with the latest `start_date` wins. The losing span is not represented in the cell colour but still appears in the week card when that cell is clicked.

### Residence Colour Derivation

Residence is not a family in the API. It is derived from spine events with `milestone_type: "relocation"`. The `location.label` string on the relocation event becomes the residence label, carried forward to all subsequent weeks until the next relocation event.

Colour is generated by hashing the label string to a hue:

```js
function locationHue(label) {
  let hash = 0;
  for (const ch of label) hash = (hash * 31 + ch.charCodeAt(0)) & 0xffffffff;
  return Math.abs(hash) % 360;
}

function residenceColor(label) {
  const h = locationHue(label);
  // --residence-l is defined in the CSS token system:
  //   light mode: 55%
  //   dark mode:  45%
  return `hsl(${h}, 40%, var(--residence-l))`;
}
```

Fixed saturation (40%) keeps residence colours visually subordinate to the richer family colours. The same label always produces the same hue across sessions and devices.

Two new tokens are required in the CSS token system:

```css
:root {
  --residence-l: 55%;
}
@media (prefers-color-scheme: dark) {
  :root {
    --residence-l: 45%;
  }
}
[data-theme="dark"] { --residence-l: 45%; }
[data-theme="light"] { --residence-l: 55%; }
```

### API Constraints

Spine events with `milestone_type: "relocation"` must always have a `location` field set. The frontend treats these as residence anchors; a relocation event without a location is silently ignored for grid colouring purposes.

Only the primary life spine (`line_key: "spine"`) is used as a source of relocation events. Secondary spines, if present, are ignored entirely by the grid — their events do not contribute to residence derivation or any other aspect of the WeekMap pre-computation.

## Data Pre-computation

At load time, after the API response is normalised (shared with the subway map), the grid runs a single pre-computation pass to build a `WeekMap` — a flat lookup keyed by `YYYY-Wnn`:

```js
// Result structure per cell:
weekMap["2023-W11"] = { family: "travel",     eventId: "evt_002",  color: "hsl(...)" }
weekMap["2022-W30"] = { family: "employment", eventId: "evt_001",  color: "hsl(...)" }
weekMap["2010-W01"] = { family: "residence",  label: "Edinburgh, UK", color: "hsl(...)" }
// null = neutral, cell not clickable
weekMap["1990-W01"] = null
```

Algorithm per week:
1. Collect all travel spans where `start_date ≤ week_end AND end_date ≥ week_start`. If multiple, take the one with the latest `start_date`.
2. Else collect all employment spans covering that week. Take first.
3. Else collect all education spans covering that week. Take first.
4. Else look up the most recent spine relocation event on or before the week's start date. Use its `location.label`.
5. Else `null`.

This runs once on load. The grid renderer reads only from the WeekMap — no per-cell logic during render.

## Rendering

The grid is rendered as plain HTML, not SVG. CSS Grid is a better fit for a uniform rectangular layout with no curves.

Each row is an independent grid container:

```html
<div class="week-grid">
  <div class="grid-row" data-year="2023">
    <span class="row-label">2023</span>
    <div class="week-cell" data-week="2023-W01" style="background: hsl(...)"></div>
    <div class="week-cell" data-week="2023-W02" style="background: hsl(...)"></div>
    <!-- ... -->
  </div>
  <!-- one .grid-row per year -->
</div>
```

Cell sizing:

| Viewport | Cell size | Gap |
|---|---|---|
| Desktop (> 768px) | 14×14px | 2px |
| Mobile (< 480px) | 8×8px | 1px |

Neutral cells (`weekMap` entry is `null`) have no background colour set and are not clickable.

---

## Click Behaviour

Clicking a non-null cell opens a week card. The card receives the ISO week key and collects all events overlapping that week across all families:

- Span events where `start_date ≤ week_end AND end_date ≥ week_start`
- Point events where `date` falls within the week

Events are grouped by family in the card (Travel, Employment, Education, Books, Fitness & Health, Film & TV, Hobbies, Spine milestones). Each item is tappable and opens its individual detail card (the same cards used by the subway map).

On desktop the week card floats near the clicked cell. On mobile it is the same bottom sheet used by the subway map.

---

## Integration with the Subway Map

| Concern | Detail |
|---|---|
| Navigation | Selecting Year zoom on the existing zoom control switches to the grid; selecting Day or Month returns to the subway map |
| Data | Single API fetch on load, shared between both views |
| Cards | Same card system, same CSS, same bottom sheet on mobile |
| Theme | Same `--color-*` token system; grid responds to light/dark mode automatically |
| Zoom state | Switching to the grid does not affect the subway map's zoom state; switching back restores it |

The grid has no zoom levels of its own.

## File Structure Additions

```
/app
  /css
    grid.css          — week grid layout, cell sizing, row labels, responsive breakpoints
  /js
    grid.js           — WeekMap pre-computation, grid DOM rendering, click handler
  /tests
    unit/
      grid.test.js
    integration/
      grid.spec.js    — desktop and mobile Playwright tests
```

`grid.js` imports the normalised API response from `api.js` (same module used by the subway map). It does not import from `timeline.js`, `lanes.js`, or `lines.js` — the two views are independent renderers over shared data.

## Testing

### Unit Tests — `grid.test.js` (Vitest)

- WeekMap assigns travel colour when a travel span covers the week
- WeekMap assigns employment colour when no travel span is active
- WeekMap assigns education colour when neither travel nor employment is active
- WeekMap assigns residence colour derived from the most recent relocation event
- WeekMap returns null for weeks before any known residence
- WeekMap returns null for weeks before birth date
- Concurrent travel spans: latest `start_date` wins
- Residence carries forward from relocation event until the next relocation event
- `locationHue()` returns the same value for the same label string on repeated calls
- `locationHue()` returns different values for different label strings
- Weeks after current ISO week are not present in WeekMap

### Integration Tests — `grid.spec.js` (Playwright)

**Desktop (1280×800)**
- Grid renders with one row per year from birth year to current year
- Birth year row has neutral cells before the birth week
- Current year row has no cells beyond current week
- Cell colour matches expected family priority for a known week in mock data
- Clicking a cell with events opens a week card listing all events that week, grouped by family
- Clicking a neutral cell does nothing
- Week card contains a tappable item that opens an individual event detail card
- Selecting Year zoom switches to the grid view; selecting Day or Month returns to the subway map with zoom state preserved

**Mobile (390×844)**
- Grid fits within viewport width with no horizontal overflow
- Cells are at minimum 8×8px
- Clicking a cell opens a bottom sheet week card
- Bottom sheet dismisses on swipe down or close tap
